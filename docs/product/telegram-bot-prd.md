# PRD: Telegram-бот для транскрибации и суммаризации медиа

## 1) Цель и ценность
Создать Telegram-бота, который принимает медиафайлы пользователя (voice/audio/video/file), автоматически формирует транскрипт и summary, а также поддерживает повторную суммаризацию уже полученного транскрипта по новому запросу пользователя.

**Ценность для пользователя:** быстро получить структурированное содержание длинного аудио/видео без ручного прослушивания.

## 2) Область MVP
В рамках MVP бот должен:
- Принимать сообщения типов: `voice`, `audio`, `video`, `document` (файл с поддерживаемым аудио/видео контентом).
- Запускать асинхронную обработку файла.
- Возвращать:
  - транскрипт;
  - краткое summary;
  - результат повторной суммаризации по дополнительному пользовательскому запросу.

## 3) Пользовательские сценарии

### Сценарий A: Отправка voice/audio/video/file
**Как пользователь**, я отправляю боту медиафайл, **чтобы** получить текст и summary.

**Основной поток:**
1. Пользователь отправляет в чат `voice`/`audio`/`video`/`file`.
2. Бот валидирует тип и длительность/размер.
3. Бот подтверждает приём и сообщает, что задача поставлена в очередь.

**Критерии приёмки (успех):**
- Бот принимает все 4 указанных типа сообщений без падения.
- При валидном файле пользователь получает подтверждение старта обработки.
- При невалидном файле пользователь получает понятное сообщение об ошибке (тип/лимит), без внутренней технической информации.

### Сценарий B: Получение транскрипта
**Как пользователь**, я хочу получить транскрипт загруженного медиа.

**Основной поток:**
1. После обработки бот отправляет транскрипт сообщением (или серией сообщений при большом объёме).
2. Если транскрипт большой, бот сохраняет читаемую структуру и порядок частей.

**Критерии приёмки (успех):**
- Транскрипт возвращается для валидного и успешно обработанного файла.
- Текст транскрипта не содержит служебных ошибок/трейсов.
- Для длинного транскрипта части приходят в корректном порядке и без пропусков.

### Сценарий C: Получение summary
**Как пользователь**, я хочу сразу после транскрипта получить краткое summary.

**Основной поток:**
1. После готовности транскрипта бот отправляет summary.
2. Summary содержит ключевые мысли и основные выводы.

**Критерии приёмки (успех):**
- Summary отправляется автоматически после транскрипта (без повторной загрузки файла).
- Summary короче исходного транскрипта и отражает его содержание.
- В ответе нет чувствительных технических данных.

### Сценарий D: Повторная суммаризация
**Как пользователь**, я хочу запросить «пересуммаризируй короче/по пунктам/с фокусом на X» на основе уже полученного транскрипта.

**Основной поток:**
1. Пользователь отправляет команду/текст с параметрами новой суммаризации.
2. Бот использует существующий транскрипт (без повторного распознавания исходного медиа).
3. Бот возвращает новую версию summary.

**Критерии приёмки (успех):**
- Повторная суммаризация выполняется без повторной загрузки файла пользователем.
- Новый ответ учитывает пользовательскую инструкцию (длина, формат, фокус).
- Время ответа повторной суммаризации меньше или равно времени полного первичного цикла обработки.

## 4) Нефункциональные требования
- **Максимальная длительность файла (MVP):** до **120 минут** аудио/видео.
- **Целевое время ответа (P95):**
  - подтверждение приёма файла: до **3 секунд**;
  - первичный результат (транскрипт + summary): до **5 минут** для файла 60 минут;
  - повторная суммаризация: до **30 секунд**.
- **Допустимая очередь задач:** до **1000** задач в ожидании без деградации корректности обработки (приоритет — сохранность и последовательность выполнения).
- **Поддерживаемые языки (MVP):** русский и английский.

## 5) Политика хранения данных
- **Исходные медиафайлы:** хранятся до **24 часов** после завершения обработки, затем удаляются автоматически.
- **Транскрипты:** хранятся **30 дней** для возможности повторной суммаризации и истории.
- **Summary:** хранятся **30 дней** вместе с привязкой к соответствующему транскрипту.
- **Удаление по запросу пользователя:** при явном запросе все связанные данные удаляются внеочередно (best effort в пределах технических ограничений очереди удаления).

## 6) Метрики успеха MVP
- Доля успешно завершённых задач транскрибации: **>= 95%**.
- Доля успешно завершённых запросов повторной суммаризации: **>= 98%**.
- Доля задач, уложившихся в целевое P95 время: **>= 90%**.
